# Functional Requirements

## Overview

This document defines the functional requirements, entity models, workflows (state machines), processors, criteria, and API behavior for the TaskManager system. It updates and consolidates the previously generated requirements to ensure logic is consistent and unambiguous.

Key rules summarized:
- POST (create) endpoints return only a JSON object with a single field `technicalId` in the response body and MUST set an appropriate Location response header (HTTP 201 Created). Creation MUST also emit a creation event that drives initial orchestration processors.
- All stored entities expose GET by `technicalId` to retrieve current persisted state. GET responses include the stored application result (including `technicalId` and the business `id` when present).
- PUT/PATCH/DELETE mutate stored entities, persist changes, and may emit events that drive workflow transitions.
- Periodic jobs and asynchronous event handlers are used for scheduled checks (due dates, retention/archival, etc.).

Notes on identifiers:
- technicalId: system-managed unique identifier used for API surface (opaque to callers). Always present and returned by POST responses.
- id: business identifier (e.g., "PRJ-001", "TASK-001"). It may be supplied by the client at creation or generated by the system if omitted.

All date/time fields use ISO 8601 strings in UTC (e.g., 2025-02-01T12:00:00Z).

## Entities

### Project

Fields:
- technicalId: String (system-assigned opaque id for API and persistence)
- id: String (business identifier for the project, optional on create; system may generate)
- name: String (project title)
- description: String (project description and goals)
- startDate: String (ISO 8601 date, optional)
- endDate: String (ISO 8601 date, optional)
- status: String (lifecycle status; see allowed values below)
- ownerId: String (reference id for the project owner)
- createdAt: String (ISO 8601 datetime when created)
- updatedAt: String (ISO 8601 datetime when last updated)
- completedAt: String (ISO 8601 datetime when marked completed; optional)
- metadata: Object (freeform key/value map for custom attributes)

Allowed project status values (canonical): created, planning, active, in_progress, completed, archived, failed, cancelled

### Task

Fields:
- technicalId: String (system-assigned opaque id for API and persistence)
- id: String (business identifier for the task, optional on create; system may generate)
- projectTechnicalId: String (reference to parent project by technicalId)
- projectId: String (reference to parent project business id if available)
- title: String (task title)
- description: String (task description)
- status: String (lifecycle status; see allowed values below)
- assigneeId: String (reference id to the person assigned; may be null)
- dueDate: String (ISO 8601 date; may be null)
- priority: String (priority label, e.g., low, medium, high)
- dependencies: Array(String) (list of task technicalIds that must complete before this task starts)
- createdAt: String (ISO 8601 datetime when created)
- updatedAt: String (ISO 8601 datetime when last updated)
- completedAt: String (ISO 8601 datetime when marked completed; optional)
- metadata: Object (freeform key/value map for custom attributes)

Allowed task status values (canonical): pending, assigned, blocked, in_progress, in_review, completed, cancelled, overdue

Behavioral notes:
- Dependencies are references to other tasks' technicalIds. Non-existent dependency ids are treated as blocking until resolved and should generate a warning/notification.
- The system stores both technicalId references and business id references where available to ease user-facing integrations.

## Workflows (State Machines)

General rules:
- Transitions can be triggered by manual user actions, asynchronous processors invoked by events, or scheduled processors.
- Processors must persist updated entities, set `updatedAt` (and `completedAt` when appropriate), and emit appropriate domain events.
- Processors and criteria should be idempotent: repeated invocations must not produce inconsistent states.

### Project Workflow (states and transitions)

Summary of states:
- created (initial on create)
- planning (team populates tasks/milestones)
- active (project accepted/activated)
- in_progress (active work ongoing; at least one non-blocked task in progress or aggregate progress > 0)
- completed (all completion criteria satisfied)
- archived (archived for retention)
- failed (critical blockers exceeded threshold)
- cancelled (manually cancelled)

Transitions and triggers (canonical):
1. Created (automatic on POST) -> Planning (StartPlanningProcessor triggered automatically by ProjectCreatedEvent)
2. Planning -> Active (ActivateProjectProcessor; usually manual, may be triggered automatically if an "autostart" flag is set)
3. Active -> In Progress (ProjectEnterInProgressProcessor; automatic when at least one non-blocked task enters in_progress or aggregated progress > 0)
4. In Progress -> Check Completion (ProjectCompletionCriterion evaluated automatically on task completion events or periodic evaluation)
5. Check Completion -> Completed (ProjectCompleteProcessor when completion criterion satisfied)
6. Completed -> Archived (ProjectArchiveProcessor, manual or automated by retention policy)
7. In Progress -> Failed (ProjectFailureCriterion when critical failure counters or blocker timers exceed thresholds)
8. Any state -> Cancelled (manual)
9. Any terminal state -> Archived (ProjectArchiveProcessor)

Project state diagram (conceptual):

CREATED --> PLANNING --> ACTIVE --> IN_PROGRESS --> COMPLETED --> ARCHIVED
IN_PROGRESS --> FAILED --> ARCHIVED
ANY_STATE --> CANCELLED --> ARCHIVED

Processor responsibilities (high-level):
- StartPlanningProcessor: initialize default milestones/metadata, set status to planning
- ActivateProjectProcessor: validate prerequisites and set status to active
- ProjectEnterInProgressProcessor: detect first real work and set project to in_progress
- ProjectCompletionCriterion: evaluate whether the project should transition to completed
- ProjectCompleteProcessor: set status completed, set completedAt, persist and notify
- ProjectFailureCriterion: evaluate persistent severe blockers and mark as failed when appropriate
- ProjectArchiveProcessor: set status archived, persist, and run cleanup (move files, release resources)

### Task Workflow (states and transitions)

Summary of states:
- pending (initial on create)
- assigned (has assignee)
- blocked (dependencies incomplete or other blocker)
- in_progress (work started)
- in_review (work submitted for review)
- completed (done)
- cancelled (manually cancelled)
- overdue (due date past; this is a derived state/flag and may coexist in metadata rather than replace status depending on implementation)

Transitions and triggers (canonical):
1. Pending (automatic on POST)
2. Pending -> Assigned (AssignTaskProcessor, manual)
3. Assigned -> Dependency Check (TaskDependencyCriterion, automatic on assignment or start); if dependencies incomplete -> Blocked
4. Blocked -> Assigned (when dependencies resolved)
5. Assigned -> In Progress (assignee starts work; manual)
6. In Progress -> In Review (SubmitForReviewProcessor, manual)
7. In Review -> Complete Check (ReviewCriterion, automatic)
8. Complete Check -> Completed (TaskCompleteProcessor if review passed) or back to In Review (if review failed)
9. In Progress -> Completed (TaskCompleteProcessor, manual or automatic)
10. Any non-terminal -> Cancelled (CancelTaskProcessor, manual)
11. Periodic -> Overdue flag/notification (TaskDueDateMonitorProcessor; this processor should not silently delete or finalize tasks but may set status or metadata depending on policy)

Task state diagram (conceptual):

PENDING -> ASSIGNED -> (DEPENDENCY_CHECK -> BLOCKED | IN_PROGRESS) -> IN_REVIEW -> COMPLETED
Any state (assigned/in_progress/pending) -> CANCELLED

Processor responsibilities (high-level):
- TaskDependencyCriterion: ensure dependencies are completed; if any dependency missing or not completed return false and mark as blocked/notify
- AssignTaskProcessor: set assigneeId, set status assigned, persist, emit TaskAssignedEvent, notify assignee
- SubmitForReviewProcessor: set status in_review, persist, notify reviewers
- TaskCompleteProcessor: set status completed, set completedAt, persist, emit TaskCompletedEvent (or ProjectCompletionEvent), notify stakeholders
- CancelTaskProcessor: set status cancelled, persist, emit TaskCancelledEvent, notify stakeholders
- TaskDueDateMonitorProcessor: scheduled job that marks tasks as overdue in metadata or status (policy-dependent) and emits overdue notifications; it should skip tasks already completed or cancelled

Important behavior rules:
- Dependencies in Task.dependencies are by technicalId. Missing dependency references should be treated as blocking and generate a warning; processors should not silently ignore missing dependencies.
- TaskDependencyCriterion should evaluate prior to allowing in_progress transitions. It should be idempotent and resilient to races (e.g., a dependency may complete concurrently).
- TaskCompleteProcessor must emit an event consumed by ProjectCompletionCriterion to re-evaluate project-level completion.

## Processors and Criteria (Detailed)

Conventions:
- All processors and criteria are executed in response to events or scheduled triggers.
- Processors must persist changes and emit domain events after successful application.
- Criteria return boolean and must be deterministic for the same inputs.

ProjectCompletionCriterion
- Purpose: Check aggregated state of tasks for this project.
- Behavior: The project is considered complete only if there is at least one non-cancelled task and all non-cancelled tasks for the project are in `completed`.
- Pseudo:
```
boolean evaluate(Project p) {
    List<Task> tasks = TaskRepository.findByProjectTechnicalId(p.technicalId);
    // Exclude cancelled tasks from completion calculation
    List<Task> relevant = tasks.stream()
        .filter(t -> !t.status.equals("cancelled"))
        .collect(Collectors.toList());
    if(relevant.isEmpty()) return false; // require at least one real task to auto-complete
    return relevant.stream().allMatch(t -> t.status.equals("completed"));
}
```

ProjectCompleteProcessor
- Purpose: Mark project as completed, set completedAt, trigger notifications and downstream archiver scheduling.
- Pseudo:
```
void process(Project p) {
    if(p.status.equals("completed")) return; // idempotent
    p.status = "completed";
    p.completedAt = Instant.now().toString();
    p.updatedAt = p.completedAt;
    ProjectRepository.save(p);
    EventBus.emit(new ProjectCompletedEvent(p.technicalId));
    NotificationService.notifyProjectCompleted(p.technicalId);
}
```

StartPlanningProcessor
- Purpose: Initialize default milestones/metadata when project created.
- Pseudo:
```
void process(Project p) {
    if(!p.status.equals("created")) return;
    initializeDefaultMilestones(p.technicalId);
    p.status = "planning";
    p.updatedAt = Instant.now().toString();
    ProjectRepository.save(p);
}
```

ProjectEnterInProgressProcessor
- Purpose: Move project to in_progress when real work starts (first task in_progress or aggregated progress threshold).
- Pseudo:
```
void process(Project p) {
    if(p.status.equals("in_progress") || p.status.equals("completed")) return;
    if(TaskRepository.existsByProjectAndStatus(p.technicalId, "in_progress")) {
        p.status = "in_progress";
        p.updatedAt = Instant.now().toString();
        ProjectRepository.save(p);
        EventBus.emit(new ProjectEnteredInProgressEvent(p.technicalId));
    }
}
```

ProjectFailureCriterion
- Purpose: Detect persistent critical failures/blockers.
- Behavior: This should be configurable; e.g., use metadata.failureCount and metadata.blockerTimestamps; when thresholds exceeded return true.
- Pseudo:
```
boolean evaluate(Project p) {
    Integer failures = (Integer) p.metadata.getOrDefault("failureCount", 0);
    return failures >= Config.projectFailureThreshold;
}
```

TaskDependencyCriterion
- Purpose: Ensure all dependency task ids are completed.
- Pseudo:
```
boolean evaluate(Task t) {
    for(String depTechId : t.dependencies) {
        Task dep = TaskRepository.findByTechnicalId(depTechId);
        if(dep == null) {
            // treat missing dependency as blocking; optionally notify
            NotificationService.notifyMissingDependency(t.technicalId, depTechId);
            return false;
        }
        if(dep.status.equals("cancelled")) {
            // cancelled dependencies can be considered satisfied or blocking depending on policy; choose blocking by default
            return false;
        }
        if(!dep.status.equals("completed")) return false;
    }
    return true;
}
```

AssignTaskProcessor
- Purpose: Assign an assignee to a task and set status to assigned. Emit TaskAssignedEvent and notify the user.
- Pseudo:
```
void process(Task t, String assigneeId) {
    t.assigneeId = assigneeId;
    t.status = "assigned";
    t.updatedAt = Instant.now().toString();
    TaskRepository.save(t);
    EventBus.emit(new TaskAssignedEvent(t.technicalId, assigneeId));
    NotificationService.notifyUser(assigneeId, "Assigned to task: " + t.id);
}
```

TaskCompleteProcessor
- Purpose: Mark task completed, set completedAt, emit events to drive project completion evaluation, and notify stakeholders.
- Pseudo:
```
void process(Task t) {
    if(t.status.equals("completed")) return;
    t.status = "completed";
    t.completedAt = Instant.now().toString();
    t.updatedAt = t.completedAt;
    TaskRepository.save(t);
    EventBus.emit(new TaskCompletedEvent(t.technicalId, t.projectTechnicalId));
    NotificationService.notifyTaskCompleted(t.technicalId);
}
```

TaskDueDateMonitorProcessor
- Purpose: Scheduled job that detects overdue tasks and updates metadata/status and sends notifications.
- Pseudo:
```
void process() {
    List<Task> overdue = TaskRepository.findWhere(dueDate < now() && status not in ("completed","cancelled"));
    for(Task t : overdue) {
        // do not downgrade a task that is already completed/cancelled
        if(t.status.equals("completed") || t.status.equals("cancelled")) continue;
        // policy: mark metadata.overdue = true and optionally set status to overdue
        t.metadata.put("overdue", true);
        // keep original status (e.g., in_progress) and optionally set a derived status flag
        t.updatedAt = Instant.now().toString();
        TaskRepository.save(t);
        NotificationService.notifyOverdue(t.technicalId);
    }
}
```

SubmitForReviewProcessor
- Purpose: Transition task to in_review and notify reviewers.
- Pseudo:
```
void process(Task t) {
    t.status = "in_review";
    t.updatedAt = Instant.now().toString();
    TaskRepository.save(t);
    EventBus.emit(new TaskInReviewEvent(t.technicalId));
    NotificationService.notifyReviewers(t.technicalId);
}
```

CancelTaskProcessor
- Purpose: Cancel a task, persist and emit TaskCancelledEvent.
- Pseudo:
```
void process(Task t, String reason) {
    if(t.status.equals("cancelled")) return;
    t.status = "cancelled";
    t.metadata.put("cancelReason", reason);
    t.updatedAt = Instant.now().toString();
    TaskRepository.save(t);
    EventBus.emit(new TaskCancelledEvent(t.technicalId));
    NotificationService.notifyTaskCancelled(t.technicalId);
}
```

## API Endpoints (EDA rules applied)

Base URL: https://api.taskmanager.com/v1
Authentication: Authorization: Bearer <token>

Design rules and principles (explicit):
- POST endpoints must return only the field `technicalId` in the response JSON body and must set the Location header pointing to the resource GET URL. Typical response is HTTP 201 Created.
- POST operations MUST emit an event representing the creation to drive orchestration (e.g., ProjectCreatedEvent, TaskCreatedEvent).
- GET endpoints return the latest persisted application state for the resource.
- PUT/PATCH/DELETE persist changes and may emit events that influence workflows.
- All endpoints are eventually consistent: processors triggered by events may update state asynchronously.

Endpoints summary:

- POST /projects
  - Purpose: Create Project entity (triggers ProjectCreatedEvent which starts Project orchestration)
  - Request JSON (example):
    {
      "name": "Website Redesign",
      "description": "Redesign the corporate website",
      "startDate": "2025-03-01",
      "endDate": "2025-04-01",
      "ownerId": "user-123",
      "metadata": {}
    }
  - Response JSON (must contain only `technicalId`):
    { "technicalId": "proj-tech-0001" }
  - Behavior: Persist Project, emit ProjectCreatedEvent. StartPlanningProcessor is invoked automatically by the orchestration subsystem.

- GET /projects/{technicalId}
  - Purpose: Retrieve stored Project result by technicalId
  - Response JSON: full stored project object (includes technicalId and business id when present)

- GET /projects
  - Purpose: Optional listing of stored projects (read-only). Response is a JSON array of stored project objects.

- POST /projects/{projectTechnicalId}/tasks
  - Purpose: Create Task entity under a project (triggers TaskCreatedEvent)
  - Request JSON (example):
    {
      "title": "Design Homepage",
      "description": "Create a modern homepage design",
      "assigneeId": "user-234",
      "dueDate": "2025-03-15",
      "priority": "high",
      "dependencies": [],
      "metadata": {}
    }
  - Response JSON (only `technicalId`):
    { "technicalId": "task-tech-0001" }
  - Behavior: Persist Task, emit TaskCreatedEvent which triggers initial Task workflow processors (e.g., TaskDependencyCriterion check). If dependencies are missing or blocking, TaskDependencyCriterion will mark task as blocked and relevant notifications will be emitted.

- GET /tasks/{technicalId}
  - Purpose: Retrieve stored Task result by technicalId
  - Response JSON: full stored task object (includes technicalId and business id when present)

- GET /projects/{projectTechnicalId}/tasks
  - Purpose: Optional list tasks for a project (read-only)

- PUT /projects/{technicalId}
  - Purpose: Update project (read/write)
  - Behavior: Updates persist and may emit events if fields relevant to orchestration are changed (e.g., manual status changes, dates). Response is the full stored project JSON.

- PUT /tasks/{technicalId}, PATCH /tasks/{technicalId}, DELETE /tasks/{technicalId}, DELETE /projects/{technicalId}
  - Purpose: Update/partial update/delete stored entities (read/write). Persist changes and emit events that may drive workflow transitions (e.g., deletion or cancellation emits TaskCancelledEvent/ProjectCancelledEvent). Deletions should be soft-deletes where possible (set status cancelled/archived) to preserve audit trails.

Important note: POST create endpoints MUST return only `technicalId` in the response body. GET responses provide full objects.

## Events and Processing Summary

Events emitted on persistence and events consumed by processors (non-exhaustive):
- ProjectCreatedEvent -> StartPlanningProcessor (automatic initialization)
- TaskCreatedEvent -> TaskDependencyCriterion evaluation and other initial startup processors
- TaskAssignedEvent -> potentially triggers TaskDependencyCriterion and notifications
- TaskCompletedEvent -> triggers ProjectCompletionCriterion evaluation and ProjectEnterInProgressProcessor if appropriate
- TaskCancelledEvent -> may change project completion evaluation
- ProjectCompletedEvent -> triggers notifications and scheduling archiver

Event bus requirements:
- The event bus should guarantee at-least-once delivery and be idempotently processed by processors to handle duplicates. Implementation choices (Kafka, RabbitMQ, etc.) are infra decisions.

Periodic processors (examples):
- TaskDueDateMonitorProcessor (detect overdue tasks and notify)
- ProjectRetentionArchiveProcessor (archive projects after retention policy)
- TaskProgressMonitorProcessor (aggregate / report task progress)

Processing guarantees:
- Processors and criteria must be idempotent and resilient to reprocessing.
- Events that change entity state should be applied in a way that preserves causality; if ordering cannot be guaranteed by transport, processors must check persisted state before applying transitions.

## Appendix

Status enums (canonical):
- Project: created, planning, active, in_progress, completed, archived, failed, cancelled
- Task: pending, assigned, blocked, in_progress, in_review, completed, cancelled, overdue

Auditing and timestamps:
- createdAt and updatedAt MUST be set on create and update.
- completedAt is optional and set when entity is moved to a completed state.

Error handling:
- Processors must log errors, mark transient exceptions for retry, and escalate persistent domain-level errors by recording them in entity metadata and notifying responsible users.
